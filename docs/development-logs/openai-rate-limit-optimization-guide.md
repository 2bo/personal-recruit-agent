# OpenAI API レート制限最適化ガイド

## 概要

このドキュメントは、MastraフレームワークベースのPersonal Recruit AgentにおいてOpenAI APIのレート制限問題を解決するために実施したプロンプト最適化プロセスをまとめたものです。

## 背景と問題

### 初期状況

- **システム構成**: Mastraフレームワーク + 3つのAIエージェント（ChecklistAgent、RecruitAgent、JobMatcherAgent）
- **実行環境**: AWS Lambda Container Image + EventBridge cron実行
- **処理対象**: 1日1回、10件の求人を並列分析
- **使用モデル**: OpenAI GPT-4.1-mini

### 発生した問題

**OpenAI APIレート制限エラー**:

```
Rate limit reached for gpt-4.1-mini: Limit 200000, Used 200000
```

**実行結果**:

- 成功: 2件の求人分析（100%マッチング）
- 失敗: 8件の求人分析（すべてレート制限エラー）
- concurrency: 1（順次実行）にも関わらず制限に到達

## 原因分析

### トークン使用量の詳細調査

1. **JobMatcherAgentプロンプトサイズ**: 16,830文字（約16KB）
2. **推定トークン消費**: 15,000-20,000トークン/求人
3. **10件分析の総消費量**: 150,000-200,000トークン
4. **APIレート制限**: 200,000トークン/分

### 問題の根本原因

**異常に肥大化したプロンプト**:

- 過度に詳細な評価基準（320行）
- 重複する説明とルール
- 冗長な例外処理ロジック
- 複数条件評価の詳細手順
- 巨大なワーキングメモリテンプレート

## 最適化プロセス

### 第1段階: 激進的削減（80%削減）

**削減前**: 16,830文字  
**削減後**: 3,314文字  
**削減率**: 80.3%

**削減内容**:

- 重複説明の統合
- 評価基準の表形式圧縮
- ワーキングメモリの簡素化
- 冗長な例示削除

**問題点**: 精度に影響する重要要素も削除

### 第2段階: 段階的復元（適切なレベルまで）

**最終サイズ**: 5,195文字  
**最終削減率**: 69.1%

**復活させた重要要素**:

1. **複数条件の個別評価プロセス**
2. **詳細な数値基準**
3. **推測禁止の具体例**
4. **一貫性確保ルール**
5. **calculation_basis記載要件**

### 第3段階: 役割明確化

**問題**: JobMatcherAgentが「分析」と「判定」の両方を担当

**解決策**: 役割分離

- **JobMatcherAgent**: 純粋な適合率分析（0-100%）
- **filterMatchingResults**: 80%以上のフィルタリング

**具体的変更**:

```typescript
// 修正前
4. 80%以上の場合のみマッチと判定

// 修正後
4. 適合理由または不適合理由を明確に説明
```

## 技術的解決策

### プロンプト最適化の具体的手法

1. **重複ルールの統合**

   ```
   修正前: 各セクションで推測禁止を個別説明
   修正後: 一箇所にまとめて参照形式
   ```

2. **評価基準の圧縮**

   ```
   修正前: 長文での詳細説明
   修正後: 数値基準を表形式で圧縮
   ```

3. **ワーキングメモリの簡素化**

   ```
   修正前: 35行の詳細テンプレート
   修正後: 5行のシンプルテンプレート
   ```

4. **役割の明確化**
   ```
   修正前: 分析 + 判定 + フィルタリング
   修正後: 分析のみ（判定は別ステップ）
   ```

### 精度維持のための配慮

**保持した重要要素**:

- 複数条件の個別評価プロセス
- 推測禁止の具体例
- 一貫性確保の数値基準
- 計算根拠の記載要件

**簡素化した要素**:

- 冗長な説明文
- 重複するルール記述
- 過度に詳細なワーキングメモリ

## 代替案の検討

### Geminiモデルへの変更

**試行内容**:

```typescript
// GPT-4.1-mini → Gemini 2.5 Flash
model: MODELS.GEMINI_2_5_FLASH;
```

**期待効果**: 無制限トークンでレート制限回避

**結果**: エラーが発生したため断念

**最終選択**: GPT-4o-miniで継続

## 学んだこと・ベストプラクティス

### 1. プロンプト設計の原則

**避けるべき設計**:

- 一つのエージェントに複数の責務を持たせる
- 冗長で重複の多い指示文
- 過度に詳細すぎる例外処理ルール
- 巨大なワーキングメモリテンプレート

**推奨する設計**:

- 単一責任の原則を適用
- 簡潔で明確な指示文
- 重要な部分のみ詳細化
- 必要最小限のメモリテンプレート

### 2. トークン使用量の最適化

**測定方法**:

```bash
# プロンプトサイズの確認
wc -c src/mastra/agents/job-matcher-agent.ts

# 削減率の計算
echo "scale=1; (16830-5195)/16830*100" | bc
```

**最適化指標**:

- プロンプトサイズ: 目標50-70%削減
- 機能維持: 重要要素は必ず保持
- 役割分離: 一つのエージェント = 一つの責務

### 3. レート制限対策

**段階的アプローチ**:

1. **プロンプト最適化**: 最も効果的
2. **モデル変更**: Geminiなど無制限モデル
3. **バッチサイズ削減**: 一度に処理する件数を減らす
4. **実行間隔調整**: より長い間隔での実行

**OpenAI API制限の理解**:

- GPT-4.1-mini: 200,000トークン/分（使用量に応じて変動）
- リクエスト数制限も存在
- 順次実行でもトークン/分の制限は有効

### 4. Mastraエージェント設計

**役割分離の重要性**:

```typescript
// 良い例
JobMatcherAgent: 適合率分析のみ;
filterMatchingResults: フィルタリングのみ;

// 悪い例
JobMatcherAgent: 分析 + 判定 + フィルタリング;
```

**プロンプト構造化**:

- ## セクション分けで可読性向上
- **重要事項**の強調
- 具体的な数値基準の明示

### 5. デバッグとモニタリング

**トークン使用量の追跡**:

- プロンプトサイズの定期チェック
- API レスポンスでのトークン使用量確認
- CloudWatch Logsでのエラー監視

**段階的デプロイ**:

- プロンプト変更後は必ずテスト実行
- 精度への影響を確認
- 段階的に最適化レベルを調整

## まとめ

OpenAI APIのレート制限問題は、以下の主要な対策により解決できました：

1. **プロンプト大幅最適化**: 69%削減でトークン使用量を大幅削減
2. **役割の明確化**: エージェントの責務を分離して複雑性を削減
3. **重要要素の保持**: 精度に影響する要素は必ず維持
4. **段階的アプローチ**: 激進的削減→復元→最適化の3段階

この最適化により、Mastraフレームワークの全機能を維持しながら、APIレート制限内での安定動作を実現できました。
